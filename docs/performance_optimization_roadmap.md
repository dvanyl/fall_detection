# NPU/CPU 性能压榨路线图

> 当前状态：25 FPS（真实环境+手机视频均稳定）  
> NPU负载：Core0: 25%, Core1: 23%, Core2: 25%  
> CPU负载：~30%

---

## 一、当前瓶颈分析

### 耗时分解（每帧 ~39ms）

| 阶段 | 耗时 | 占比 | 瓶颈类型 |
|------|------|------|----------|
| 摄像头读取 | ~5ms | 13% | I/O |
| RGA预处理 | ~8ms | 21% | 硬件加速 |
| NPU推理 | ~10ms | 26% | NPU |
| CPU后处理 | ~3ms | 8% | CPU |
| 绘制+跌倒检测 | ~8ms | 21% | CPU |
| MJPEG编码 | ~5ms | 13% | CPU |
| **总计** | **~39ms** | **100%** | **25.7 FPS** |

### 关键观察
- NPU总负载仅 ~75%（3×25%），还有 ~25% 余量
- CPU负载 ~30%，还有 ~70% 余量
- 管线没有明显的单一瓶颈，属于**均衡负载**

---

## 二、压榨路线（按收益/难度排序）

### 第1级：低难度，中等收益（预期 +3-5 FPS）

#### 1.1 降低输入分辨率
```
当前：640×640（模型输入）
优化：480×480 或 320×320
效果：NPU推理时间减少 40-75%，但精度会下降
难度：需要重新量化模型
```

#### 1.2 使用 FP16 代替 INT8
```
当前：INT8 量化模型
优化：FP16 模型（RK3588 NPU 原生支持 FP16）
效果：精度提升，但推理时间可能略增
适用：如果对精度要求更高
```

### 第2级：中难度，高收益（预期 +5-8 FPS）

#### 2.1 真正的异步管线（流水线并行）
```
当前：读取→推理→绘制→推流 是串行的
优化：3个阶段完全解耦，用队列连接
  读取线程 → [帧队列] → 推理线程池 → [结果队列] → 绘制+推流线程
效果：各阶段可以重叠执行，理论FPS = max(读取FPS, 推理FPS, 绘制FPS)
难度：需要重构线程模型
```

#### 2.2 RGA硬件加速BGR2RGB
```
当前：cv::cvtColor(BGR2RGB) 使用CPU
优化：用RGA硬件做颜色空间转换
效果：减少CPU开销，RGA做resize的同时做颜色转换
```

#### 2.3 NPU核心负载均衡调度
```
当前：固定绑定 Core0/1/2
优化：动态调度，谁空闲谁处理
效果：避免某个核心因RGA失败而空闲
实现：用原子变量跟踪各核心状态
```

### 第3级：高难度，高收益（预期 +10-15 FPS）

#### 3.1 零拷贝预处理
```
当前：摄像头帧 → cv::Mat → RGA处理 → memcpy到tensor
优化：使用DMA缓冲区，摄像头帧直接映射到NPU输入
效果：消除所有内存拷贝，减少 ~5ms
难度：需要修改RKNN内存管理，使用RKNN_FLAG_MEM_ALLOC_OUTSIDE
```

#### 3.2 硬件JPEG编码
```
当前：cv::imencode (CPU软件编码)
优化：使用RK3588的硬件JPEG编码器（VEPU/JPEG模块）
效果：编码时间从5ms降到<1ms
难度：需要直接操作硬件编码器API
```

#### 3.3 模型量化优化（QAT/混合精度）
```
当前：全INT8量化
优化：关键层用FP16，非关键层用INT8
效果：在精度和速度之间找到更好的平衡点
```

### 第4级：极限压榨（预期 40+ FPS）

#### 4.1 双模型流水线
```
当前：3个NPU核心各跑1个完整模型
优化：将模型拆成2个阶段，Stage1在Core0+1，Stage2在Core2
效果：利用NPU内部流水线，减少单帧延迟
```

#### 4.2 自定义NPU算子
```
优化：将后处理(NMS)也放到NPU上执行
效果：减少CPU-NPU数据传输
```

## 三、验证方法

```bash
# NPU负载监控
cat /sys/kernel/debug/rknpu/load

# CPU负载监控
top -bn1 | grep "Cpu(s)"

# 帧率监控（程序自带日志）
# 每30帧输出一次 FPS

# 内存监控
free -h
cat /proc/meminfo | grep -i cma
```

## 四、建议

**当前25 FPS已经是一个很好的成绩**。如果要继续压榨：

1. **短期**（1-2天）：尝试降低输入分辨率到480×480，看精度损失是否可接受
2. **中期**（1周）：实现真正的异步管线（流水线并行）
3. **长期**（2周+）：零拷贝预处理 + 硬件JPEG编码

**注意**：FPS提升的边际效益递减。从17→25是质的飞跃（+47%），从25→30只是+20%，投入产出比下降。
